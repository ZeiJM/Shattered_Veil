PRIMARY GOAL:
Add a safe arena targeting layer so skills and attacks can begin showing range, target zones, and area shapes on the big battle map.

This pass should make the arena feel more tactical, but should not fully rebalance every skill yet.

DESIGN DIRECTION:
The future battle system should support large arena battles with:
- variable map sizes and shapes
- tactical movement
- range-based skills
- area-of-effect shapes
- line, cone, burst, self, aura, zone, and global targeting
- destructible objects
- rare terrain bonuses
- Veilbreak field effects
- Field Clashes
- cinematic battle log entries

IMPLEMENTATION REQUIREMENTS:

1. Add skill range metadata support.

   Every battle action and Veil Magic skill should be able to have metadata such as:

   {
     range: number,
     shape: "single" | "line" | "cone" | "burst" | "self" | "aura" | "zone" | "global",
     radius: number,
     targetType: "enemy" | "ally" | "self" | "tile" | "emptyTile" | "object" | "any",
     requiresLineOfSight: boolean,
     canTargetObjects: boolean,
     terrainInteraction: string | null
   }

   Do not break skills that do not yet have this metadata.
   Add safe defaults.

2. Add default range/shape assignment helpers.

   Add helper logic that can infer safe temporary metadata for existing skills.

   Suggested defaults:
   - Basic Strike: range 1, shape single, target enemy
   - Flurry: range 1, shape single, target enemy
   - Guard: range 0, shape self
   - Self buffs: range 0, shape self
   - Heals/support: range 4, shape single or burst, target ally/self
   - Projectiles: range 5, shape single or line
   - AoE magic: range 4, shape burst, radius 1 or 2
   - Veilbreaks: global or large zone, but visual-only for now
   - Boss-wide mechanics: global
   - Unknown skills: range 4, shape single, target enemy

   This should be a temporary foundation, not final balance.

3. Add arena range calculation helpers.

   Add or update helper functions such as:
   - getTilesInRange(position, range, arena)
   - getLineTiles(origin, directionOrTarget, range, arena)
   - getConeTiles(origin, target, range, angleOrWidth, arena)
   - getBurstTiles(center, radius, arena)
   - getAuraTiles(origin, radius, arena)
   - getGlobalTiles(arena)
   - getValidTargetTiles(caster, skill, arenaState)
   - getAffectedTiles(caster, targetTile, skill, arenaState)
   - getUnitsOnAffectedTiles(affectedTiles, arenaUnits)
   - hasLineOfSight(origin, target, arenaState)

   Keep line-of-sight simple for now.
   Blocking terrain/objects can be respected if already available, but do not overcomplicate it.

4. Add visual targeting modes to the ArenaBoard.

   When the player selects an action or skill:
   - highlight valid target tiles
   - highlight affected area tiles
   - show enemy/ally/object tokens that would be affected
   - show out-of-range targets differently if safe
   - show a simple tooltip with range, shape, and target type
   - keep movement preview separate from skill targeting preview

   Required visual states:
   - valid movement tile
   - valid target tile
   - affected area tile
   - selected target tile
   - out-of-range tile
   - blocked line-of-sight tile, if implemented
   - occupied tile
   - destructible object tile

   Keep the card clean.
   Do not overload the UI with too much text.
   Use compact badges or small labels.

6. Add target preview before confirming an action.

   For player actions:

   - clicking a valid enemy/tile should preview affected tiles
   - player should be able to confirm or cancel
   - existing direct button behavior should still work if targeting mode fails

   Important:

   Use a safe fallback to existing target selection.

7. Soft range enforcement.

   - If metadata or positions are missing, fall back to the current combat targeting behavior.
   - Do not make existing combat impossible because of missing data.

   Add TODO comments for full hard enforcement in a later pass.
