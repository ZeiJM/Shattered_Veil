==================================================
SECTION A — CLEAN TARGETING MODEL
==================================================

1. Add a unified target model for all combat actions.

Every battle action/skill/item/tactical action should support a target profile like:

{
  targetKind: "self" | "enemy" | "ally" | "anyUnit" | "ground" | "emptyGround" | "object" | "field" | "arenaWide",
  shape: "single" | "line" | "cone" | "burst" | "wall" | "ring" | "aura" | "zone" | "self" | "arenaWide",
  range: number,
  radius: number,
  requiresLineOfSight: boolean,
  canTargetSelf: boolean,
  canTargetAllies: boolean,
  canTargetEnemies: boolean,
  canTargetGround: boolean,
  canTargetObjects: boolean,
  intent: "damageEnemy" | "debuffEnemy" | "healAlly" | "supportAlly" | "selfBuff" | "movement" | "field" | "object" | "mixed"
}

If existing skills already have range/shape metadata, adapt it rather than duplicating it.

2. Add target validation helpers.

Create or update helpers such as:

- getActionTargetProfile(action)
- getActionIntentType(action)
- getDisplayRangeTag(action)
- getValidTargetTiles(action, caster, arenaState)
- getValidTargetUnits(action, caster, arenaState)
- getAffectedTiles(action, caster, target, arenaState)
- getAffectedUnits(action, caster, target, arenaState)
- validateActionTarget(action, caster, target, arenaState)
- canActionTargetUnit(action, caster, targetUnit)
- canActionTargetTile(action, caster, targetTile)
- clearActionTargetingState()

Target validation rules:
- Enemy-targeting actions must require clicking/tapping an enemy token or a tile containing an enemy.
- Ally-targeting actions must require clicking/tapping an ally token or a tile containing an ally.
- Self-targeting actions must allow clicking/tapping the player’s own token.
- Self-only actions may also show a clear “Use on Self” confirmation button.
- Ally-support actions may target self unless the action explicitly excludes self.
- Ground actions may target terrain tiles.
- Empty-ground actions may target only empty walkable tiles.
- Object actions may target destructible objects.
- Arena-wide or Field actions should not require a normal unit click unless specifically designed.

3. Make token clicking reliable.

The player should not need to guess whether to click a tile or a unit.

Requirements:
- Enemy token click selects enemy target for enemy-targeting actions.
- Ally token click selects ally target for ally-targeting actions.
- Player token click selects self for self-targeting/self-buff actions.
- If the player clicks an invalid target, show a clear short message and do not execute.
- Empty tile clicks should not accidentally target enemies unless the action is ground/empty-ground/movement/field based.
- Mobile taps should work.
- Desktop clicks should work.
- Long press/hover should show details, not immediately execute.

4. Add a targeting state machine.

Use a clear targeting flow:

IDLE
→ ACTION_SELECTED
→ TARGET_PREVIEW
→ TARGET_CONFIRMED
→ RESOLVE_ACTION
→ CLEAR_TARGETING

The battle UI should show:
- selected action
- valid targets
- affected area
- confirm/cancel controls where needed
- reason if target is invalid

Do not instantly fire complex actions without showing preview unless the action is simple self-only or explicitly safe.

- targetBinding: "chosenTarget" | "caster" | "affectedEnemies" | "affectedAllies" | "affectedUnits" | "affectedTiles" | "fieldArea"
- timing: "instant" | "startTurn" | "endTurn" | "onHit" | "onMove" | "onFieldTick" | "onExpire"

10. Add effect target inheritance.

Effects should not all blindly apply to the clicked unit.

Rules:
- If effect targetBinding is "chosenTarget", apply to the selected target.
- If targetBinding is "caster", apply to the caster, even if the action targeted an enemy.
- If targetBinding is "affectedEnemies", apply only to enemies in the affected area.
- If targetBinding is "affectedAllies", apply only to allies/self in the affected area.
- If targetBinding is "affectedUnits", apply to all valid units in area.
- If targetBinding is "affectedTiles", apply to ground/terrain.
- If targetBinding is "fieldArea", apply to the field/zone.

Example:
A strike can damage an enemy while also applying Veilflare Focus to the caster.
A heal can restore an ally while also applying a small self cooldown or focus effect.
A field action can affect tiles while also buffing the caster.

11. Add friendly-fire rules.

Each effect should support:

friendlyRule: "all" | "alliesOnly" | "enemiesOnly" | "selfOnly" | "notSelf"

Rules:
- positive effects default to allies/self unless specified
- negative effects default to enemies unless specified
- mixed effects must define their own targetBinding/friendlyRule
- avoid accidentally healing enemies or damaging allies unless an action explicitly says so

SECTION D — TARGET PREVIEW AND FORESHADOWING
==================================================

12. Add clean preview colors.

When an action is selected, show targeting foreshadowing before use:

- Blue/teal = movement range and movement destination
- Red = enemy damage or enemy debuff target/affected area
- Green = ally/self healing, shielding, cleanse, buff, support
- Purple/gold = Veilbreak, Field, Field Clash, arena-wide effects
- Orange = object/destructible targeting
- Grey = invalid, blocked, out of range

13. Preview requirements.

When selecting an action:
- valid targets should glow
- affected tiles should be shown
- affected units should be highlighted
- invalid tiles should be muted/grey
- hover/tap target should preview the exact area
- canceling action clears preview
- confirming action clears preview after resolution

Specific behavior:
- Steady Strike: clicking enemy shows red single-target preview.
- Flurry Strike: clicking enemy shows red single-target preview.
- Damaging Veil Magic: enemy or area shows red preview.
- Heal/support skill: ally/self shows green preview.
- Tactical Step/Move: tiles show blue/teal preview.
- Veilbreak/field action: arena or field zone shows purple/gold preview.
- Object-breaking action: destructible objects show orange preview.

14. Add preview summaries.

When hovering/long-pressing a valid target, show a compact preview card:

- action name
- target name/tile
- range tag
- shape
- likely affected units
- effect summary
- confirm/cancel options on mobile if needed

Do not show exact final damage if that is hard, but show expected intent clearly.

SECTION F — BATTLE RESOLUTION PIPELINE
==================================================

18. Clean up action resolution.

Use this pipeline:

1. Player selects action.
2. System reads target profile.
3. System calculates valid target tiles/units.
4. Player clicks/taps enemy/self/ally/tile depending on action.
5. System validates target.
6. System calculates affected tiles.
7. System calculates affected units.
8. System applies effect tags based on targetBinding.
9. System applies costs/cooldowns/action use.
10. System writes battle log entries.
11. System clears preview and advances turn/action state.

19. Add strong safety fallbacks.

The system must not crash if:
- action has missing target profile
- action has missing effect tags
- caster position is missing
- target position is missing
- ally list is empty
- enemy list is empty
- action targets self but self token is not found
- action targets enemy but target tile has no enemy
- area effect hits no one
- terrain/object data is missing
- old skills still use old metadata
- save data has old action structures

Use safe defaults.
